Introduction

Welcome to your last Exercise on JavaScript! In the previous Exercise, you learned all the basic language syntax and rules of JavaScript—which was a considerable amount of new information to take in—but now that you’ve powered through the difficult part, you should have a solid basis for all your scripting needs. Congratulations!

In this Exercise, you’ll take a look at jQuery, the most famous JavaScript library, and how it can help you add interactivity to your website. You’ll also be looking at how you can apply what you’ve learned to your portfolio website by changing your menu into a more mobile-friendly version for small viewports. Let’s get started!

What is jQuery?

You may have heard of jQuery in passing conversations and meetings. For many people, it’s become somewhat synonymous to JavaScript, but what is it really?

In short, jQuery is a JavaScript library. When we talk about a “library” in the context of programming, it means a set of pre-written code that we can use in our own projects. Some libraries are fairly small and offer only very specific things, while others offer a plethora of functions and utilities to use. jQuery falls into the latter category as it provides hundreds of different functions that cover a large variety of topics.

Though jQuery can be used for a variety of different things, there are three main functionalities people tend to use it for:

DOM Traversal & Manipulation

In Exercise 2.1: Introduction to JavaScript, we talked about the DOM (Document Object Model), which is a representation of the HTML page and what allows you to interact with the elements on the screen. While all of this is possible without jQuery, it’s often more convenient to do it with jQuery.

Event Handling

In programming, it’s often necessary to react to an action the user has just taken. This is called event handling. An event refers to anything that happens while a user is visiting your webpage, and, in JavaScript, you can react to these user actions. Below are just some of the available events:

  • click: when a user clicks on an element
  • submit: when a user submits a form
  • resize: when the window is resized
  • scroll: when the user scrolls
  • keydown/keyup: when the user presses or releases a key
  • mouseover/mouseout: when a user moves their mouse over or out of an element
  • change: when the value of an input changes

TIP!
For a complete list of events you can use, refer to this event reference list by MDN.
As with DOM traversal and manipulation, event handling can also be done without jQuery, but it’s just easier to do it with. We’ll talk about why in a later section.

Ajax

The term Ajax, short for asynchronous JavaScript and XML, refers to loading data with JavaScript (asynchronously and in the background) without causing an entire webpage to refresh. With traditional HTML pages, the browser loads a full HTML document from the server every time the user presses a link or button. With Ajax, however, you can send and receive data from a server and display it to the user without loading a brand-new HTML page. This allows for better (and cooler!) interactivity.

Since a full discussion of Ajax exceeds the scope of what we want to accomplish in this course, we’ll be primarily focusing on DOM manipulation and event handling. If you’re looking for more information on Ajax, though, feel free to take a look at this general introduction or into the Ajax documentation of jQuery.

How to Use jQuery

One of the reasons jQuery is so popular is that it’s extraordinarily easy to use. All you need to do is load the script. You can do this in several ways.

Download it here (use the latest compressed production version) and then place the file in your “/js” folder where you also have the scripts.js file. Then, simply include it in your HTML the same as you would your scripts.js (refer to image below). Make sure that jquery-3.3.1.min.js is added before the scripts.js—scripts are loaded and executed in the order they’re included in the HTML, so you always want to load your jQuery before your main scripts.js doc.

<script src="js/jquery-3.3.1.min.js"></script>
<script src="js/scripts.js"></script>

A better (and more advisable) way is to link to a hosted jQuery file on a CDN as it’s more efficient and will most likely arrive faster to your user than loading it from your own server. Go to this page and get the link for jQuery Core 3.3.1 and use the minified version. The link you get here should be added in the same way as you would do for the downloaded version, looking something like this:

<script src="https://code.jquery.com/jquery-3.3.1.min.js" integrity="sha256-FgpCb/KJQlLNfOu91ta32o/NMZxltwRo8QtmkMRdAu8=" crossorigin="anonymous"></script>
<script src="js/scripts.js"></script>

Note!
Instead of downloading jQuery and embedding it in your project, it’s common practice to load it from a content delivery network (CDN). A CDN is a geographically distributed group of servers that work together to provide fast delivery of Internet content—meaning jQuery will load faster for your users. (It will either be downloaded from a server close to the location of the user or your browser will simply reuse a jQuery version that’s already been downloaded from a different website and is currently saved in your browser’s cache). Check out this page for more information on CDNs.

What’s a “compressed” or “minified” version of a library? These are versions where whitespace and comments have been stripped out, leading to a smaller file size—which, in turn, means your users will need to download less data when visiting your page. If you’re curious about what minified jQuery looks like, check it out here and compare it to the normal version.

That’s all it takes to use jQuery! Now, you can use all the functions it provides in your scripts.js.

The Ups and Downs of jQuery

We can go on and on about how great jQuery is, but before we dive deeper into working with it, let’s take a moment to discuss why you would or wouldn’t use jQuery.

While using it comes with many benefits, it does have one major downside—it requires all your users to load the full jQuery library before your page can do anything. On a slower Internet connection, this could mean a noticeable impact on your page load times. With this in mind, you should always consider whether you actually need jQuery. For most of JavaScript’s functionality, there are ways to get what you want using only the so-called “Vanilla JavaScript,” in other words, JavaScript without any libraries.

Apart from its ease of use, another main advantage of jQuery is its handling of browser incompatibilities. Since the early days of the Internet, some browsers have supported certain features better or worse than others. A number of years ago, Internet Explorer 6 (and subsequently 7 and 8) was the bane of many web developers’ lives as it lacked support for features compared to other browsers.

Since JavaScript is a scripting language that’s executed on your users’ machines, you have no control over which browser they use to view your page. This means that if you want users with certain browsers to be able to view your page, you can’t use features their default browsers don’t support.

With the birth of jQuery, developers could use exciting new features without worrying about those complicated edge cases. Since that time, however, browsers across the board have gotten much better. Nowadays, most things you need can be handled without using jQuery; for example, it might be reasonable (depending on your target group of users) to only support Internet Explorer 10 and up. This would mean you can use all language features of JavaScript that IE10 and newer support. There’s even a resource, You Might Not Need jQuery, that allows you to select which version of IE you need to support to see what the jQuery-equivalent functions would look like.

It’s quite likely that you’ll encounter JavaScript libraries at some point. Since jQuery is often considered an important and foundational JS library, we’ll stick with jQuery in this Exercise for the sake of hands-on practice and familiarity. For future projects, however, keep in mind that when only adding a few of lines of functionality, you can probably avoid including jQuery in your site.

DOM Traversal & Manipulation

When working with JavaScript, you’ll mostly be working with DOM nodes. A DOM node (often just referred to as simply a “node”) is any HTML element on the page; some examples include a <button>, a <div> or an <input>. There are many things you can do with DOM nodes, but first you need to select them with JavaScript.

Selecting Nodes

With jQuery, you select nodes the same way you select elements to style them with CSS! Check out this example:

var input = $("input");
var containers = $(".container");
var listItems = $(".list > li");

Selecting elements like this results in a jQuery collection, which is simply an array of jQuery elements. A collection can be empty if you use a selector where no element is found on the page, or it can have any number of elements. After creating a jQuery collection, you can try playing around with it, for example:

var oldItems = $(".old-item");
oldItems.remove(); // Remove all found elements from the page

var greenItems = $(".green-item");
greenItems.css("color", "green"); // Make the text color green

var hideItems = $(".hide-item");
hideItems.hide(); // Don't remove them, just hide them
hideItems.show(); // Now show them again!

These are just a few examples of what you can do with the DOM. When we do something with a DOM node, we call this DOM manipulation—like the oldItems.remove() function in the example above. You can find a list of things you can do with jQuery collections in the jQuery Documentation.

One important thing to keep in mind is that all these functions work the same no matter how many items are in your selection. If there’s no item in the selection, nothing will happen. If there are three in it, all three of them will be manipulated in the same way.

Traversing Nodes

Another thing jQuery is very good at is traversing nodes, which involves moving from one node to another within a data structure. For example, say you have a variable with a reference to a list (<ul>). Now, you can easily tell all items on the list to do the same thing:

var list = $("ul");
var listItems = list.find("li");
listItems.hide();

// Or, you can do this all in one line!
$("ul").find("li").hide();
// Which is the same as...
$("ul > li").hide();

You can also try something a bit more complex; for example:

$("input[disabled]").parent().addClass("disabled-container");

This would select all inputs with the attribute disabled, get their parents, then add the class disabled-container to them. Here is a list of functions for traversing the DOM with jQuery.

Event Handling

The second big jQuery usage you’ll be utilizing in your own portfolio website is event handling. In JavaScript, you bind events to DOM nodes; for example, you might bind the click event to a button. An event then gets what’s called an “event listener.” This is a function that’s called every time the event happens:

$(".button").on("click", function() {
  alert("This button has been clicked!");
});

Now, anytime an element with the class button is clicked, an alert will appear. The function in an event handler has a few special properties, the most important of which is the ability to get the element that triggered the event:

$(".button").on("click", function() {
  $(this).hide();
});

With $(this) inside of an event handler, you get the HTML node that triggered the event. In this example, all elements with the button class would get this event listener, but often we need to know exactly which button was clicked—this is what the $(this) tells us.

Other times, we may want to change something when a user hovers over an element:

$(".item").on("mouseover", function() {
  $(this).parent().addClass("is-hovered");
});
$(".item").on("mouseout", function() {
  $(this).parent().removeClass("is-hovered");
});

These are just a few examples of what you can do with jQuery events. Together with DOM traversal and manipulation, event handling can help you realize a wide range of cool effects and interactivity.

Project Exploration

Now, let’s add some interactivity to our portfolio website! We’ve already created a great-looking portfolio site that’s responsive and works well; however, what if you wanted a real mobile menu for mobile devices that 1) is displayed when the hamburger icon is selected, and 2) appears as a normal menu when the site is viewed on larger screens?
Let’s see if we can implement this using HTML, CSS, and Javascript.

Mobile Menu Toggle

What do you need to create a mobile menu that changes to a normal menu on larger screen sizes using Javascript? First, you’ll need to make some changes in your code.

You’ll want to start by making a space for a hamburger icon in your <header>. To do that, you’ll probably need to change the width of your logo. For the logo on our example wireframes, we’ll need to change it (in the default CSS) to max-width: 150px;. Be careful, though, as this will also change the size of our logo for all our other screen sizes, as well! We can prevent this by adding back the original, larger value for the logo size in the media query for min-width: 641px. This will ensure the size of the logo won’t change on larger screen sizes (since there won’t be a hamburger menu on those screen sizes).

Next, you’ll need to add a <div> in your <header> that allows you to toggle the mobile menu on and off. This <div> will have the class navtoggle and the ID myNavtoggle. (Don’t forget to close your new <div> after the closing </nav> tag!)

The complete code would look like this:

<header>
  <div class="container">
    <div class="navtoggle" id="myNavtoggle">
      <img class="logo" src="img/logo.svg" alt="John Doe UX UI logo" >|
      <nav>

The next step is to add the actual hamburger menu icon. If you followed the Tip! in Exercise 1.4 that linked you to Fontawesome for your social media icons, then this will be very simple. If not, go ahead and visit Fontawesome now and copy-paste the link for the CDN into the <head> of each one of your pages.

Once you’ve done that, add an extra list item at the end of your menu after the contact link. This will be where you place your hamburger menu icon. The complete list item should look something like this:

<li><a href="javascript:void(0);" class="icon" onclick="menuToggle()"><i class="fas fa-bars"></i></a></li>

What does all of that mean?

The href has a value of javascript:void(0);. The void(0) operator is a short, simple script that evaluates to undefined, essentially telling the browser to stay on the same page rather than reload the page.

We also added a class, icon, so that we can target this list item in CSS.

The onclick="menuToggle()" is what will activate the JavaScript we add later so that the menu icon actually does something. You can probably see already that the menu will be activated “on click.”

Finally, the <i class="fas fa-bars"></i> is the hamburger icon from Fontawesome.
Now, if we check the results in our browser, the <header> would look something like this:

What do you notice here?

On mobile, as you have a hamburger menu now, your navigation items should be hidden unless the menu is clicked.
Also, you should remove the center class for the <header> so that your logo will be on the left and your menu icon on the right.

For tablet and larger, you need to hide the menu icon.

How do we do all this? Why, in our media queries, of course! Let’s take a look.

The first step is to remove the text-align center we set in the default CSS styling for the <header> selector. You can simply delete that line since we don’t need it anymore.

Next, it’s time to add some code to one of our media queries. You originally had the following rule-set in your default CSS:

nav > ul > li {
  display: inline-block;
}

Now, however, you need to move this out of your default CSS and into your first media query with min-width: 641px. This code was what was making your navigation items appear side-by-side. On mobile, however, you no longer want this (as your navigation items need to be vertically stacked in your menu). You do, however, still want this on larger screen sizes that won't have a menu. This is why you need it in your media query but not in your base code.

The mobile menu should now look like this:

Next, you need to hide the navigation items (except the menu icon). To achieve, this you’ll need the following code:

.navtoggle li:not(:last-child) {
  display: none;
}

What it does is hide (display: none;) all list elements (<li>) except for the last one (:last-child is pseudo code that targets the last “child” of it’s parent element). This will hide all the list elements except for the hamburger menu icon.

Again, since you’re adding this to your base code, you’ll want to add something to your media query to ensure it doesn’t happen on larger screen sizes (since there won’t be a menu at all!). For this, add the following code, which will reverse what you just did, in the media query for min-width: 641px:

.navtoggle li:not(:last-child) {
  display: inline-block;
}

Now, the navigation items will only be hidden on mobile (but will be visible on larger screen sizes). While you’re here in the media query, you can also add some code that will hide the hamburger menu icon. For this, you’ll want to add:

.navtoggle li:last-child {
  display: none;
}

Once you’ve added this, your <header> on tablet/desktop should return to what it looked like at the beginning of this Exercise. Your mobile <header>, however, should look something like this:

Now, it’s time to position our menu icon to the right of the <header> and align it with the logo. To do this, we use the position property, which we explained earlier when discussing how you can set your <header> position to fixed so that it doesn’t scroll.

This is where the extra <div> we added will come in handy. Set the position of this new <div> to relative in your base code. It would end up looking like this:

.navtoggle, .navtoggle.responsive {
  position: relative;
}

You’ll notice that we also added another selector for .navtoggle with the class .responsive attached. This isn’t for use in conjunction with your HTML (because, as you’ll remember, we didn’t add that class in our HTML) but, rather, for use with JavaScript later. This is the class we’ll use to select our menu using JavaScript and style our menu.

Go ahead and check what your <header> looks like in your browser now. Likely, nothing has changed, as everything we just did was simple behind-the-scenes “staging,” so to speak. Let’s go ahead and position the hamburger menu icon and make it a little bigger, too. Our default font is already set to 16px, so let’s give it a new font-size that’s a little bit bigger:

.navtoggle li .icon {
  position: absolute;
  right: 0;
  top: 2px;
  font-size: 20px;
}

After adding this styling in your CSS, your <header> should look something like this:

You’re probably noticing a few things at this point. Notice that white gap between the <header> and your hero element? Also, the hamburger menu item doesn’t look quite in place. It could stand to move over to the right a little bit more. Lastly, there’s a little bit too much space underneath the logo. It should be centered more vertically within the <header>.

Let’s start with the hamburger menu icon. To resolve this issue, we can simply remove the padding for the right side of the icon. To the code we just added for .navtoggle li .icon, simply add: padding-right: 0;, which should align it properly.

Next, we’re going to solve the space below the image. For this, the easiest way is to use the Web Inspector. If you hover over the <nav> element in the Web Inspector, you’ll see the nav light up like this:

It seems as though our blank navigation menu is what’s causing this extra space. This is fairly easy to solve, as we can simply remove the <nav> element from our base code. Go ahead and delete this code in your CSS:

nav {
  display: inline-block;
  width: 100%;
}

This solves part of the problem, but there’s still a little bit of extra space. Still in the Web Inspector, hover over the next element, the <ul>, and you’ll notice there’s some top and bottom margin there from the default browser settings.

Since these margins are simply taking up space, let’s remove them for our base code (though we still want them to appear on larger screens!) We already have this code in our CSS file for our base code:

nav > ul {
  list-style-type: none;
  padding: 0;
}

Now, simply add margin: 0;. Before you forget, however, go back into your media query for min-width 641px screen sizes and add the 1em top and bottom margins back (since you still want those margins on your larger screens). It should look like this in your CSS:

nav > ul {
  margin: 1em 0;
}

Once this is done, your <header> on mobile should look something like this:

There’s still a tiny bit of extra spacing below the logo that’s caused by the line-height. Rather than try and remove that, however, it would be easier to simply adjust the padding of the <header> so that the space above and below the logo is the same. In our example, if we change the padding for the header selector in the base code to padding: 10px 0 4px;, it should look relatively even.

The last thing we need to inspect is the large white gap between our <header> and the hero section. (You’ll only see this on your own site if you followed along in the previous Achievement to create a fixed header.) Remember how we added padding for the <main> element to push our content down, adjusting for the height of the header? Let’s do that again here. Use the Web Inspector to find the height of your <header> element. In our example, it’s just a bit over 56px. Next, give the main selector a new padding-top of 56px to account for the size of the <header>, like this:

main {
  padding-top: 56px;
}

After refreshing, your mobile site should look like this:

Great! The <header>’s looking pretty good now. But there’s still one problem: the menu doesn’t actually work. That’s because this needs to be done in JavaScript.

Go ahead and open your “scripts.js” file in Atom. Remember above when we added the code onclick="menuToggle()” to our menu? What this is saying is that when a user clicks on the hamburger icon (onclick), the browser should launch the menuToggle(); function. What does the menuToggle(); function do? Let’s write it now! Declare a new function in your “scripts.js” file like so:

function menuToggle() {
}

In this function, we want to be able to change the class of the menu so that it shows the menu links when we click on the icon. This is done by first assigning the menu object (using the ID) to a variable:

function menuToggle() {
  var x = document.getElementById('myNavtoggle');
}

The above code searches the document for the element with the ID you provide. The ID we’ve provided here is myNavtoggle, which is the ID that we gave the new <div> that we added to our navigation. Thus, our navigation object would be assigned to the new variable x.

Next, we need to change the class of this object based on whether it’s already shown or not (if the menu’s already shown, we want it to disappear on click; if it’s not shown, we want it to appear on click). In order to show the menu, the classname responsive has to be added to the classname of the menu. See what we’ve done below:

function menuToggle() {
  var x = document.getElementById('myNavtoggle');
  x.className += ' responsive';
}

Note that you'll need the space before responsive for successful execution of your code.

The above code adds the responsive classname to the className of x (accessed through dot notation). But what about if the menu is already shown? In this case, we would need to hide the menu. To solve this problem, we need an if-else statement checking to see if the menu is already shown or not (and we only want to add responsive to the classname if the menu isn’t already shown). Take a look below:

function menuToggle() {
  var x = document.getElementById('myNavtoggle');
  if (x.className === 'navtoggle') {
    x.className += ' responsive';
  } else {
    x.className = 'navtoggle';
  }
}

Can you see the if-else statement? In the if section, the browser checks if the className of x is navtoggle. If it is, it adds responsive to className (which will show the menu). In the else section, if the className of x is NOT navtoggle (because it’s currently both navtaggle responsive), it will change className back to simply navtoggle (effectively hiding your menu).

Save your scripts file, open your CSS, and, in the base code section, where you have all of your styling for the navigation, add:

.navtoggle.responsive li  {
  display: block;
}

If you’ll remember, the menu was previously hidden by CSS because of the .navtoggle li:not(:last-child) { display: none;} we added, which effectively hid all the menu items except for the hamburger menu icon. By changing the display property to a value of block when the .responsive class is toggled, it will override our previous code and show the complete menu.

Now, when you select an item from the menu, you’ll navigate to that page and the menu will automatically close. When you select “contact,” however, though it will navigate to the contact section, the menu won’t close. This is because the “contact” link is an anchor link. To resolve this, add onclick="menuToggle()" to the “contact” link. This will activate the JavaScript for this anchor and automatically close the menu when selected.

Our menu is working properly now. All that’s left to do is give it some extra styling so that it will look better (and be easier to tap). For our example site, we used the following rule-sets to do this (but you’re free to play around and create your own to get the exact feel you want for your menu). As you’ll notice, they’ve all been added in the responsive section that shows when the menu icon is toggled:

.navtoggle.responsive nav {
  padding: 24px 0 0;
}
.navtoggle.responsive li  {
  display: block;
  padding: 10px 0;
  border-top: 1px solid rgba(255,127,80,0.5);
}

.navtoggle.responsive li:last-child  {
  padding: 0 0 10px;
}

After adding these (or a combination of your own), test your menu out again in your browser. It should be looking pretty slick!

Other JavaScript Libraries & jQuery Plugins

There are many ways to interact with your projects and jQuery. Feel free to browse the jQuery documentation, where you can find many different ways for interacting with your DOM.

Apart from jQuery, there are many other JavaScript libraries out there, as well. Recently, “single page apps” have become more and more prevalent. These are web pages that rely heavily on JavaScripts and often utilize huge libraries to enable app-like experiences in the browser. Examples for libraries that allow this are: React, Angular, Vue.js, and Ember.js.

Even apart from these, there’s still a wealth of other jQuery plugins and code snippets that provide different functionality with very little effort. These are JavaScript libraries (i.e., JavaScript files) that depend on jQuery being loaded before they themselves are loaded. You can include them the same way you include any other script (after jQuery and before your “scripts.js”!), then use them wherever you like in your code. Feel free to explore the plugins on Unheap or other similar platforms. There, you can search for specific UI interaction patterns, for example, “Modals.”
